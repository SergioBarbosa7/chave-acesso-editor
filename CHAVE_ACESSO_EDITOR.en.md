Access Key Structure (44 Digits)

The key is organized in the following sequence, according to the [integration manual](http://www.sistemaunico.com.br/arquivos/manuais/chave_nfe.pdf):
- **02 digits:** Issuer state code (UF) (example: 41 for PR, 35 for SP).
- **04 digits:** Issue year and month (YYMM).
- **14 digits:** NF-e issuer CNPJ; in case of CPF, use 11 digits with `000`.
- **02 digits:** Document model (example: 55 for NF-e, 65 for NFC-e).
- **03 digits:** Document series.
- **09 digits:** NF-e number.
- **01 digit:** Emission type.
- **08 digits:** Numeric code generated by the system.
- **01 digit:** Check digit (DV)

# Check Digit Calculation (cDV)

The **cDV** is calculated based on the **first 43 digits**, using the **modulo 11** algorithm.

## Step by Step

1. Take the **first 43 digits of the key**
2. Multiply each digit by a weight
3. The weights go from **2 to 9**, from right to left (cyclic)
4. Sum all results
5. Calculate the remainder of division by 11
6. Apply the final rule

---

## Math Rule

rest = sum % 11

if (rest == 0 or rest == 1)
    cDV = 0
else
    cDV = 11 - rest

---

## Weight Sequence

From right to left:

2, 3, 4, 5, 6, 7, 8, 9, 2, 3, ...

---

## Practical Example

### Base (43 digits)

4224111708156200010055003000005181139472620

### Calculation (summary)

(last digit × 2) +
(next digit × 3) +
(next digit × 4) +
...

(Following the weights until the first digit)

### Result

rest = 6
cDV = 11 - 6 = 5

### Final key

42241117081562000100550030000051811394726205

### Field-by-field comparison for the example

Using the key `42241117081562000100550030000051811394726205`, the correct field split is:

- `42` -> UF code
- `2411` -> Issue year and month
- `17081562000100` -> Issuer document
- `55` -> Document model
- `003` -> Series
- `000005181` -> Document number
- `1` -> Emission type
- `39472620` -> Numeric code
- `5` -> Check digit

Note: the earlier example used `13947262`, which was inconsistent with the fixed-width key structure. The correct numeric code for this sample key is `39472620`, because the preceding `1` belongs to the emission type field.

---

Implementation example

Since this project is intended for Java/Spring, here is a ready-to-use example:

```java
public static int calculateDV(String key43) {
    int sum = 0;
    int weight = 2;
    for (int i = key43.length() - 1; i >= 0; i--) {
        int num = Character.getNumericValue(key43.charAt(i));
        sum += num * weight;
        weight++;
        if (weight > 9) {
            weight = 2;
        }
    }
    int rest = sum % 11;
    if (rest == 0 || rest == 1) {
        return 0;
    }
    return 11 - rest;
}
```

# Definition

I want a simple interface where the user can enter the access key and see the definition of each field, identify the UF directly from the key, know the numeric code, and in short get a field-level visualization.

The user should also be able to change the key fields and have the check digit recalculated.

Build the backend with Java and a simple frontend with Bootstrap.

# Open Questions for Implementation

This section contains the answered scope decisions to guide implementation.

## 1. Accepted document models

Should the system accept only models `55` and `65`, or any 2-digit document model?

Answer: All 2-digit models, including NF-e, CT-e, NFC-e, etc.

## 2. CPF in the 14-digit taxpayer field

When the issuer uses CPF, how should it be represented inside the 14-digit field?

Options that needed clarification:
- Pad with `000` on the left
- Pad with `000` on the right
- Another rule

Answer: Right padding.

## 3. Key input mode

Should the user be able to:
- Enter only the full 44-digit key
- Enter only the fields separately
- Use both modes

Answer: Enter only the full 44-digit access key, with or without mask characters.

## 4. Field editing

After loading a key, will the user be able to edit:
- All editable fields individually
- Only some specific fields
- The full key and the separated fields at the same time

Answer: The full key and the separated fields at the same time.

## 5. Recalculate DV

Should the check digit be recalculated:
- Automatically when any field changes
- Only when clicking a button
- In both cases

Answer: Only when clicking a button.

## 6. Application validations

Besides validating the DV, which rules should exist?

Examples:
- Allow only numbers
- Require exactly 44 digits in the full key
- Validate UF code
- Validate month in `YYMM`
- Validate accepted models
- Validate exact length of each field

Answer: All of them except "allow only numbers", because the input may contain masks.

## 7. Interface formatting

On screen, should the key and fields appear:
- Without mask, numbers only
- With visual mask
- With an option to switch between both formats

Answer: The user may paste either format, but internally the system must process only the unmasked value.

## 8. Behavior for invalid keys

When the key is invalid, the application should:
- Block visualization entirely
- Still show the fields and highlight the errors
- Allow editing for correction

Answer: If the key has 44 characters and is invalid, the application should show which field is invalid, whether cDV, CNPJ, etc. If it does not have 44 characters, show only invalid length.

## 9. UF table

Can the implementation assume the standard NF-e UF code table to identify the state?

Answer: Yes.

## 10. Frontend style

Do you want:
- Spring Boot serving HTML pages with Bootstrap
- Spring Boot with a REST API and separate frontend in the same project

Answer: Spring Boot + Bootstrap.

## 11. Persistence

Does the application need to save history, last viewed keys, or any data in a database?

If yes, describe what should be saved.

Answer: No.

## 12. Interface language

Should the interface be fully in Brazilian Portuguese?

Answer: For now yes, but leave room for i18n.
